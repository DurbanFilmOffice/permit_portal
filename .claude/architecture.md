### Layer rules (enforced by architecture, not by linting)

```
app/ → calls services/ or repositories/ directly for simple reads via Server Components
app/api/ → calls services/ only
services/ → calls repositories/ only
repositories/ → calls db/ only
components/ → call services/ via Server Actions or fetch from app/ pages
```

Never skip a layer. A component never imports a repository. A route handler never imports Drizzle directly.

## Database schema

### `users`
```
id, email (unique), full_name, password_hash
role: enum(applicant, permit_officer, admin, super_admin) default applicant
email_verified: bool, verification_token, reset_token, reset_token_expires
created_at, updated_at
```

### `permits`
```
id, user_id FK→users, permit_type, status: enum(draft,submitted,under_review,approved,rejected,returned)
project_name, site_address, description
form_data: jsonb  ← ALL dynamic form fields go here, never as columns
submitted_at, created_at, updated_at
```

### `permit_documents`
```
id, permit_id FK→permits, file_name, file_url, file_type, file_size_bytes, uploaded_at
```

### `permit_status_history`
```
id, permit_id FK→permits, changed_by FK→users
old_status, new_status, comment (officer's optional reason — NOT the comment thread)
changed_at
```

### `permit_comments`
```
id, permit_id FK→permits ON DELETE CASCADE, author_id FK→users
body: text, created_at, updated_at
```
Rules: applicants comment only when status is submitted|under_review|returned. Officers any status. Flat list only.

### `portal_notifications`
```
id, user_id FK→users (recipient)
type: enum(comment_added, status_changed, permit_submitted, permit_approved, permit_rejected)
title: text, body: text (optional)
permit_id FK→permits (nullable, ON DELETE SET NULL)
is_read: bool default false, created_at
```
Bell behaviour: unread count = WHERE user_id = session.user AND is_read = false. Poll /api/notifications/unread-count every 30s. No WebSockets.

### `notification_recipients`
```
id, email, name, role (label only), is_active: bool default true, created_at
```
Admin manages this via UI. On permit submit → email all WHERE is_active = true.

### `email_log`
```
id, permit_id (nullable FK), recipient_email, template_name
status: enum(sent, failed, pending), error_message, sent_at
```

## Workflow tables (create in initial migration — engine not built yet)

All five tables must exist from day one. They sit empty until the engine is implemented.

### `workflow_templates`
```
id, name, permit_type (nullable — null means default for all types)
description, is_active: bool, version: int, created_by FK→users
created_at, updated_at
```

### `workflow_step_definitions`
```
id, workflow_template_id FK ON DELETE CASCADE
step_order: int, name, description
assigned_role: text (role that can claim this step)
assigned_user_id FK→users (nullable — if set, specific person; if null, any user with role)
is_required: bool default true
config: jsonb  ← { sla_hours, allow_parallel, escalate_after_hours }
created_at
```

### `workflow_instances`
```
id, permit_id FK ON DELETE CASCADE, workflow_template_id FK
current_step_order: int default 1
status: enum(pending, in_progress, completed, rejected, returned)
started_at, completed_at, created_at
```

### `workflow_step_instances`
```
id, workflow_instance_id FK ON DELETE CASCADE, step_definition_id FK
step_order: int, assigned_to FK→users (null until claimed), claimed_at
status: enum(pending, awaiting, approved, returned, skipped)
  NOTE: no 'rejected' at step level — step rejection always means 'returned to applicant'
due_at, actioned_at, created_at
```

### `workflow_decisions` (immutable — insert only, never update)
```
id, step_instance_id FK, decided_by FK→users
decision: enum(approved, returned)
comment (optional reason), decided_at
metadata: jsonb (future: IP, device, duration)
```

### Workflow confirmed decisions
- Any user with the assigned role can claim a step (no forced assignment)
- On step return → permit.status = returned, applicant edits and resubmits
- Resubmission always restarts from step 1 (previous decisions preserved as audit history)
- `rejected` is terminal, set by super admin only, bypasses workflow entirely

### Workflow open questions (answer before building engine)
- [ ] Different approval chains per permit type?
- [ ] Any parallel steps (multiple reviewers must all approve)?
- [ ] SLA per step — what happens on breach?
- [ ] Can super admin force-skip a step mid-workflow?
- [ ] Max resubmission attempts before permanent rejection?

## Permit form strategy

Form fields are NOT defined yet. When they arrive:

**Rule:** Every form field goes into `form_data jsonb`. Never create a DB column for a form field unless the system needs to filter/sort/aggregate by it in SQL.

**Promote to column (filterable):** permit_type, project_name, site_address
**Keep in jsonb (display only):** applicant ID, contractor details, GPS coords, declarations, all other fields

**Adding a field:** add to Zod schema + step component. No migration.
**Removing a field:** remove from Zod schema + component. Old records unaffected. No migration.
**Renaming a field:** add new key, remove old key in schema. Run one-off SQL UPDATE to backfill existing records. Still no ALTER TABLE.

Form UI pattern: multi-step wizard, one component per section:
```
src/components/permits/permit-form/
  index.tsx                   ← orchestrates steps + holds RHF state
  steps/step-01-[name].tsx    ← one file per form section
  permit-form-navigation.tsx  ← prev/next/submit + progress indicator
```

Zod schema location: `src/lib/validations/permit-form.schema.ts` (create when form is defined)

## Build order

```
Phase 1 — Foundation
  1. create-next-app scaffold + Biome + Drizzle config + folder structure
  2. ALL schema files (including workflow tables) → single migration
  3. Scaffold workflow.service.ts with NotImplementedError stubs

Phase 2 — Layout shell
  4. shadcn init + theme (dark first, Zinc)
  5. ThemeProvider, Sidebar (collapsible desktop + Sheet drawer mobile), Header, DashboardShell
  6. Route group layouts: (auth), (applicant), (admin) + placeholder pages + root redirect
  7. Middleware skeleton (passthrough until Auth.js is configured)

Phase 3 — Authentication
  8. Register + email verification
  9. Login + sessions + real middleware (RBAC)
  10. Forgot / reset password

Phase 4 — Citizen permit flow
  11. My applications list
  12. Submit permit (once form fields are defined)
  13. Permit detail + status timeline
  14. Edit + document upload

Phase 5 — Comments
  15. comment-thread + comment-form components
  16. comments.repository + comments.service
  17. Wire into applicant + admin detail pages

Phase 6 — Approval / Rejection
  18. approval-actions component (AlertDialog)
  19. approvePermit / rejectPermit in permits.service
  20. permit-status-update email template + portal notification

Phase 7 — Notification bell
  21. portal-notifications.repository
  22. /api/notifications route handlers
  23. notification-bell component + wire into header

Phase 8 — Admin portal
  24. All applications table (TanStack + Nuqs)
  25. Admin permit detail page
  26. User management + notification recipients config

Phase 9 — Dashboard & polish
  27. Analytics dashboard (Recharts + KPI cards)
  28. Account settings
  29. Vitest unit tests + Playwright E2E
```
